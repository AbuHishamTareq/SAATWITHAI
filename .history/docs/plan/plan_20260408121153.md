## Phase 2: Database Design & Migrations

### 2.1 Database Schema Design

Design and implement the following tables:

| Table                     | Description                                      |
| ------------------------- | ------------------------------------------------ |
| `users`                   | All user accounts (customers, providers, admins) |
| `roles`                   | User roles (via spatie/laravel-permission)       |
| `permissions`             | Role permissions                                 |
| `model_has_roles`         | User-role mapping                                |
| `model_has_permissions`   | User-permission mapping                          |
| `addresses`               | User addresses (customers & providers)           |
| `services`                | Service offerings by providers                   |
| `service_categories`      | Service categorization                           |
| `bookings`                | Service bookings                                 |
| `booking_statuses`        | Booking status history                           |
| `payments`                | Payment records                                  |
| `transactions`            | Payment transaction details (Telr)               |
| `wallets`                 | User wallet balances                             |
| `wallet_transactions`     | Wallet credit/debit history                      |
| `provider_availabilities` | Provider working hours & breaks                  |
| `provider_schedules`      | Recurring availability patterns                  |
| `ratings`                 | Service ratings                                  |
| `notifications`           | User notifications                               |
| `audit_logs`              | System audit trail                               |

## Development Order Recommendation

**Recommended execution order for parallel development:**

1. **Phase 1** → All three setups (Backend, Flutter, React) can be done in parallel
2. **Phase 2** → Backend only (database is foundational)
3. **Phase 3** → All three in parallel (auth is critical for all apps)
4. **Phase 4** → All three in parallel (service CRUD)
5. **Phase 5** → All three in parallel (booking core)
6. **Phase 6** → All three in parallel (payments)
7. **Phase 7** → Backend + Flutter in parallel
8. **Phase 8** → Backend + Flutter in parallel
9. **Phase 9** → Backend only
10. **Phase 10** → React CRM only
11. **Phase 11** → All three in parallel
12. **Phase 12** → All three in parallel
